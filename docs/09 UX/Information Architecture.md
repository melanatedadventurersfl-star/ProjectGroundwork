# Information Architecture

## Project Groundwork

This document defines the initial navigation and information structure for Groundwork.

Groundwork should organize work by purpose, not by disconnected data types.

---

# Navigation Principle

Users should begin with the work they care about.

The primary navigation model should center on Workspaces.

A user should think:

> I need to work on Little Camp of Horrors.

Not:

> I need to find the right module.

---

# Top-Level Navigation

Initial top-level navigation:

1. Home
2. Workspaces
3. My Work
4. People
5. Resources
6. Insights
7. Settings

---

# Home

Home should be role-aware.

It should summarize:

- Active Workspaces
- Assigned work
- Upcoming milestones
- Items needing attention
- Recent activity
- Suggested next actions

Home should not be a dumping ground.

It should be a calm launchpad.

---

# Workspaces

The Workspaces area should show all Workspaces the user can access.

Workspaces may be filtered by:

- Type
- Status
- Organization
- Owner
- Date
- Recent activity

Each Workspace card should show:

- Name
- Type
- Status
- Owner
- Key date
- Health or readiness signal
- Next action

---

# Workspace Navigation

Inside a Workspace, recommended navigation:

1. Mission Control
2. Timeline
3. Tasks
4. People
5. Documents
6. Resources
7. Notes
8. Lessons Learned
9. Settings

Additional tabs may appear when capabilities are enabled.

Examples:

- Budget
- Inventory
- Registration
- Forms
- Insights

---

# My Work

My Work should show the user what they are personally responsible for.

It may include:

- Assigned tasks
- Team responsibilities
- Upcoming dates
- Workspaces needing attention
- Recent updates

This view should be role-aware.

---

# People

People should show contacts, members, workers, volunteers, clients, vendors, partners, and other people connected to the Organization.

A Person may have multiple roles.

People should not be duplicated just because they appear in different contexts.

---

# Resources

Resources should provide access to reusable organizational assets.

Examples:

- Documents
- Templates
- Inventory
- Venues
- Vendors
- Guides
- Saved knowledge

Resources should support reuse across Workspaces.

---

# Insights

Insights should show meaningful summaries and patterns.

Examples:

- Workspace progress
- Task completion
- Budget trends
- Resource usage
- Lessons learned
- Repeated blockers

Insights should lead to action.

---

# Settings

Settings should include configuration for:

- Organization
- People and roles
- Workspace types
- Capabilities
- Branding
- Templates
- Billing

Settings should be visible based on permission level.

---

# Design Standard

Navigation should reduce hunting.

Users should not need to remember where information lives.

The product should bring relevant information to the context where it is needed.
