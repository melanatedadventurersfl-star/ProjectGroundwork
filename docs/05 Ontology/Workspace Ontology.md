# Workspace Ontology

## Project Groundwork

This document defines the Workspace, the primary user-facing concept in Groundwork.

A Workspace is the dedicated home for meaningful work.

---

# Definition

A Workspace is a focused environment where people, resources, tasks, timelines, documents, communication, and knowledge come together around a specific purpose.

A Workspace may represent an event, campaign, client booking, operational area, asset, initiative, training program, or internal project.

---

# Purpose

The Workspace exists to give work a home.

Instead of forcing users to search across separate modules, a Workspace gathers everything needed to understand and move the work forward.

A Workspace should answer:

- What are we working on?
- Why does it matter?
- Who is involved?
- What needs attention?
- What happens next?
- What have we learned?

# Workspace Types

Workspace Types define common patterns of work.

Initial Workspace Types may include:

- Experience Workspace
- Inventory Workspace
- Marketing Workspace
- Client Booking Workspace
- Volunteer Workspace
- Product Workspace
- Training Workspace
- Partner Workspace
- Operations Workspace

Workspace Types are templates, not hard limits.

They should define useful defaults while allowing organizations to customize as needed.

---

# Core Properties

Every Workspace should include:

- Workspace ID
- Organization
- Name
- Type
- Description
- Owner
- Status
- Visibility
- Start Date
- Target Date
- Created Date
- Created By
- Last Updated
- Related System
- Enabled Capabilities

---

# Workspace Statuses

Recommended statuses:

- Draft
- Planning
- Preparing
- Ready
- Active
- Reviewing
- Archived

Not every Workspace Type needs every status.

Statuses should remain plain and easy to understand.

---

# Core Capabilities

Most Workspaces should support:

- Overview
- Tasks
- Timeline
- People
- Documents
- Communications
- Notes
- Insights
- Activity History

Additional capabilities may be enabled depending on the Workspace Type.

---

# Relationships

A Workspace may connect to:

- One Organization
- One or more Systems
- Many People
- Many Tasks
- Many Documents
- Many Communications
- Many Resources
- Many Templates
- Many Lessons Learned
- Many related Workspaces

A Workspace may also use objects owned by other Systems, such as inventory, venues, vendors, vehicles, and budgets.

---

# Actions

Common Workspace actions include:

- Create Workspace
- Edit Workspace
- Add People
- Enable Capability
- Add Task
- Add Document
- Add Communication
- Add Resource
- View Mission Control
- Review Progress
- Capture Lesson Learned
- Create Template
- Archive Workspace

---

# Mission Control

Mission Control is the main operational view of a Workspace.

Mission Control should summarize:

- Current status
- Priorities
- Timeline
- Open tasks
- Blockers
- People involved
- Recent activity
- Key documents
- Workspace health
- Suggested next actions

Mission Control should help a user understand what matters without digging.

---

# Workspace Health

Workspace Health is a summary of readiness, progress, and attention areas.

It may include signals such as:

- Timeline progress
- Task completion
- Budget status
- People coverage
- Document readiness
- Resource readiness
- Open blockers

Workspace Health should lead to action.

It should not be decorative.

---

# Templates

A Workspace can become a Template.

Templates preserve useful structure from completed work.

A Workspace Template may include:

- Default capabilities
- Timeline
- Checklist
- Roles
- Tasks
- Documents
- Budget structure
- Communication patterns
- Lessons Learned

Templates are one of the main ways Groundwork preserves institutional memory.

---

# Design Rule

A Workspace should feel like a room where everything needed for that body of work is already gathered.

The user should not need to assemble the story manually.

The Workspace is the story.
